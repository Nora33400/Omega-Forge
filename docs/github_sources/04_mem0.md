# 04 — mem0ai/mem0 Decomposition

Repository: `mem0ai/mem0`

License from source pack: Apache-2.0.

mem0 is relevant to Omega-Forge as a reference for long-term memory in agent systems. For Omega-Forge, the immediate value is not a general personal-memory agent; it is a memory layer for failures, repairs, decisions, and repeated project patterns.

## 1. Role for Omega-Forge

Current Omega-Forge can detect and report validation failures:

```text
ArtifactValidator
↓
RepairAgent
↓
repair_task
```

But it does not remember:

```text
Have we seen this error before?
What fixed it last time?
Which template caused it?
Which repair failed repeatedly?
Which repo context triggered it?
```

mem0 inspires the future:

```text
FailureKnowledgeBase
↓
RepairMemory
↓
Contextual recall
↓
Better repair proposal
```

## 2. What to study

Study memory concepts rather than copying code.

Important patterns:

```text
Memory item schema
User/project/session scoping
Add memory
Search memory
Update memory
Delete memory
Embedding-backed retrieval
Metadata filters
Confidence/relevance scoring
Memory summarization
```

Likely upstream areas to inspect after cloning:

```text
mem0/
mem0/memory/
mem0/vector_stores/

tests/

```

## 3. What could be copied

Default: copy nothing now.

Potential future copy candidates only after exact commit and license audit:

```text
Small schema ideas
Small memory operation naming conventions
Tiny examples of memory add/search UX
```

Better approach:

```text
Build a tiny Omega-native FailureKnowledgeBase first.
Only add mem0 adapter later if needed.
```

## 4. What must not be copied

Avoid copying:

```text
Full memory engine
Provider integrations
Vector store integrations wholesale
Embedding wrappers wholesale
Prompt templates wholesale
Docs wholesale
Hosted/cloud-specific assumptions
```

Reason: Omega-Forge first needs a deterministic local memory for its own CI-safe repair loop. A general agent memory framework is too broad for the current stage.

## 5. What should be adapted as original Omega-Forge code

Create original modules:

```text
omega_forge/core/failure_memory.py
omega_forge/core/memory_store.py
```

Minimal data model:

```text
FailureMemoryEntry
├─ id
├─ created_at
├─ project_type
├─ task_title
├─ template_name
├─ artifact_path
├─ error_signature
├─ error_text
├─ repair_task
├─ repair_result
├─ status: unresolved/resolved/repeated/ignored
└─ metadata
```

Minimal operations:

```text
add_failure(entry)
find_similar(error_signature, project_type=None)
mark_resolved(entry_id, repair_result)
mark_repeated(entry_id)
list_recent(limit=20)
```

Initial storage:

```text
JSON file under workspace root
```

Later storage:

```text
SQLite
Vector backend adapter
mem0 adapter
```

## 6. Adapter idea

Possible future adapter:

```text
omega_forge/adapters/mem0_adapter.py
```

Purpose:

```text
Store and retrieve repair/failure memories through mem0
when the optional dependency is installed.
```

Adapter policy:

```text
- optional dependency
- disabled by default
- local memory first
- no cloud memory by default
- no personal data storage unless explicit
```

## 7. License/compliance notes

Apache-2.0 is permissive but requires preserving license and notices when copying. It includes patent grant language.

Before copying anything:

```text
1. Clone upstream.
2. Record exact commit SHA.
3. Copy LICENSE into third_party_notices/mem0_LICENSE.
4. Check NOTICE file if present.
5. Document copied files and modifications.
6. Prefer adapter/dependency over vendoring.
```

## 8. First concrete integration task

Do not integrate mem0 directly yet.

First Omega-native task:

```text
Build FailureKnowledgeBase
```

Acceptance criteria:

```text
- Can record a validation failure.
- Can generate a stable error signature.
- Can find previous failures by signature.
- Can mark a failure as resolved.
- Can detect repeated failures.
- Uses local JSON storage first.
- Tests cover add/find/resolve/repeat.
- CI remains green.
```

## 9. Decision

Decision: **study memory schema and retrieval patterns; do not copy mem0 source code yet**.

Integration type:

```text
Original Omega-Forge failure memory first
Optional mem0 adapter later
No vendored code now
```

Priority: P0/P1.

It is P0 for autonomy because repair memory is central to self-improvement. It is P1 for implementation order because PatchPlan/PatchApplier should come first or alongside it.

## 10. Ω-Forge task proposal

Add tasks:

```text
Build FailureMemoryEntry model
Build JSON FailureKnowledgeBase
Connect RepairAgent to FailureKnowledgeBase
```

This lets Omega-Forge begin learning from its own failed generations without importing a large general-purpose memory system.
